两阶段提交机制详解

阶段1：Prepare（准备阶段）

1 InnoDB在Buffer Pool中修改数据页，将数据页标记为"脏页"

2 生成Undo Log用于事务回滚和MVCC（多版本并发控制）

3 将数据变更写入Redo Log Buffer

4 将Redo Log刷盘并标记为PREPARE状态

5 此时事务未提交，可以安全回滚

// Prepare阶段伪代码 prepare_transaction(transaction) { // 1. 在内存中修改数据页 modify_data_pages_in_buffer_pool(); // 2. 生成Undo Log用于回滚 create_undo_log(); // 3. 写入Redo Log并标记为Prepare状态 write_redo_log(PREPARE); // 4. 刷盘Redo Log flush_redo_log_to_disk(); return PREPARE_SUCCESS; }

阶段2：Commit（提交阶段）

1 MySQL Server将事务的Binlog写入Binlog Cache

2 将Binlog刷盘持久化（fsync系统调用）

3 通知InnoDB提交事务

4 InnoDB将Redo Log标记为COMMIT状态并刷盘

5 释放行锁和事务相关资源

// Commit阶段伪代码 commit_transaction(transaction) { // 1. 写入Binlog write_binlog(transaction); // 2. 刷盘Binlog flush_binlog_to_disk(); // 3. 通知InnoDB提交 innodb_commit(transaction); // 4. InnoDB标记Redo Log为Commit状态 write_redo_log(COMMIT); // 5. 释放锁和资源 release_locks_and_resources(); return COMMIT_SUCCESS; }

崩溃恢复机制

数据库崩溃重启后，InnoDB恢复过程：

1 扫描Redo Log文件，找到所有事务记录

2 对每个Prepare状态的事务，检查对应的Binlog是否存在

3 如果Binlog存在且完整，则提交事务并重做

4 如果Binlog不存在或不完整，则回滚事务

// 崩溃恢复伪代码 crash_recovery() { // 1. 扫描Redo Log transactions = scan_redo_log(); for (tx in transactions) { if (tx.status == COMMIT) { // 已经提交的事务，重做 redo(tx); } else if (tx.status == PREPARE) { // Prepare状态的事务，检查Binlog if (binlog_contains(tx.xid)) { // Binlog存在，提交并重做 commit_and_redo(tx); } else { // Binlog不存在，回滚 rollback(tx); } } } }

这种机制保证了Redo Log和Binlog的原子性一致性，即两者要么都写入，要么都不写入，确保数据一致性。
